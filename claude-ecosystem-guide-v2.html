Prompting Frameworks

Structured approaches to get the best results from Claude.

üîó Chain-of-Thought

Break complex reasoning into explicit steps. Best for math, logic, multi-step problems.

Let's think step-by-step: 1. First, identify the inputs 2. Then, calculate intermediate values 3. Finally, derive the answer

üå≥ Tree of Thought

Explore multiple reasoning paths, evaluate each, prune bad branches. Best for creative/strategic work.

Generate 3 approaches to [problem]: Approach 1: [desc] ‚Üí Evaluate ‚Üí Keep/Discard Approach 2: [desc] ‚Üí Evaluate ‚Üí Keep/Discard Expand best approach(es).

‚ö° ReAct

Interleave reasoning and tool use. Best for tasks requiring external data or multi-tool workflows.

Thought: I need to find current info Action: web_search("topic") Observation: [results] Thought: Now I can answer Final Answer: [conclusion]

üìö Few-Shot

Provide examples to establish pattern. Best for specific output formats, style matching.

Example 1: Input ‚Üí Output Example 2: Input ‚Üí Output Now you try: Input: [your input] Output: [Claude generates]

üéØ R-I-S-E

Structured prompt framework: Role, Input, Stop boundary, Emit format.

Act as [ROLE]. Given [INPUT], Stop at [BOUNDARY]. Deliver [FORMAT].

üè∑Ô∏è Prefix Tags

Set scope and rigor level with tags. [SPIKE] = fast prototype, [PROD] = production-ready.

[SPIKE] Quick prototype for... [PROD] Production-ready with tests... [SKEPTIC] Challenge all assumptions... [SKELETON] Architecture before code...

Best Practices Checklist

  • Be clear and specific
    "Debug this Python function throwing KeyError on line 42" not "Help with my code"
  • Provide context
    Include audience, purpose, constraints, and expected format
  • Use examples when needed
    Show Claude the pattern you want with 2-3 concrete examples
  • Request step-by-step for complex tasks
    Break large tasks into phases: "First design, then implement, then test"
  • Iterate and refine
    Start broad, then narrow. Ask for revisions rather than restarting

Security Best Practices

Enterprise security patterns for Claude implementations.

  • A01: Access Control (RLS)
    Row-level security verified, users can't see others' data (tested)
  • A02: Security Misconfiguration
    No defaults, buckets locked, private by default
  • A03: Injection Prevention
    All queries parameterized, no string concatenation
  • A04: Secure Design
    Rate limits in place, threat model reviewed
  • A05: Cryptographic Failures
    TLS enforced, PII encrypted at rest, no plaintext secrets in logs
  • A09: Supply Chain Security
    Deps scanned (npm audit, Snyk), lockfile committed, CI/CD secrets isolated
